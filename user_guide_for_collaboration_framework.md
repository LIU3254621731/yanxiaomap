# 多任务分工协作框架使用指南

## 概述
本指南详细说明如何使用为"考研院校地图择校网站"项目设计的多任务分工协作框架。该框架将复杂项目拆分为多个独立任务，通过多个专用Agent并行开发，使用共享状态文件进行通信和协调，最终集成为完整系统。

## 框架核心概念

### 1. Agent角色体系
框架定义了7种Agent角色，每种角色负责特定领域：

| Agent类型 | 职责 | 负责任务 | 关键技能 |
|-----------|------|----------|----------|
| database Agent | 数据库设计与数据管理 | T001 | MySQL, SQL, 数据建模 |
| backend Agent | 后端API与服务开发 | T002, T004 | Spring Boot, MyBatis-Plus, Redis |
| frontend Agent | 前端界面与交互开发 | T003 | Vue3, TypeScript, 高德地图API |
| security Agent | 安全防护与合规实现 | T005 | 安全加固, 攻击防护, 合规要求 |
| deployment_test Agent | 测试与部署实施 | T006 | Docker, 测试框架, 监控配置 |
| integration Agent | 项目集成与协调 | T007 | 项目管理, 技术协调, 风险控制 |

### 2. 通信机制：共享状态文件
所有Agent通过一个共享的Markdown文件进行通信：
- **文件位置**: `c:\Users\32546\Desktop\yanxiaomap\project_status.md`
- **更新频率**: 每2-4小时（各Agent按规范更新）
- **主要内容**: 任务状态、进度报告、接口定义、问题记录、决策日志

### 3. 任务拆解与分配
项目被拆分为7个独立任务，每个任务有明确的：
- 输入输出定义
- 验收标准
- 时间估计
- 依赖关系

## 快速开始

### 步骤1：初始化项目状态
1. 检查`project_status.md`文件是否已创建（框架已提供模板）
2. 确认所有Agent已分配任务（见任务分配状态表）
3. 设置项目启动时间，更新最后更新时间

### 步骤2：启动各Agent工作
1. **database Agent** 首先开始工作，因为其他任务依赖数据库设计
2. **backend Agent** 在数据库设计开始后可并行进行项目初始化
3. **frontend Agent** 可基于接口定义和Mock数据开始开发
4. **integration Agent** 全程监控协调

### 步骤3：监控项目进展
1. 每小时查看`project_status.md`更新
2. 关注"通信记录"章节，了解各Agent进展和问题
3. 检查"待解决问题"列表，及时提供决策
4. 查看"用户输入区"，回答Agent提出的问题

## 详细使用流程

### 1. 作为项目负责人（用户）

#### 职责
- 提供需求澄清和业务决策
- 确认技术方案和界面设计
- 解决Agent无法达成共识的争议
- 验收最终交付成果

#### 操作方法
1. **定期检查**: 每天至少检查2次`project_status.md`
2. **回答问题**: 在"用户输入区"回答Agent提出的问题
3. **提供决策**: 对需要业务决策的问题提供明确指示
4. **进度确认**: 确认里程碑完成，批准进入下一阶段

#### 关键决策点
- 接口设计确认（前后端Agent协调后需要最终确认）
- 界面设计确认（frontend Agent完成界面后需要确认）
- 技术方案争议（当Agent无法达成共识时）
- 需求变更审批（任何需求变更需要批准）
- 交付成果验收（最终验收签字）

### 2. 作为integration Agent

#### 职责
- 监控所有Agent进度
- 协调接口对齐和冲突解决
- 维护项目状态看板
- 生成项目报告

#### 操作方法
1. **每小时检查**: 检查各Agent进度更新
2. **每日站会**: 在共享文件中组织站会，要求各Agent报告：
   - 昨日完成工作
   - 今日计划工作
   - 遇到的问题
   - 需要协调的事项
3. **问题协调**: 发现不一致或冲突时，组织相关Agent讨论解决
4. **状态更新**: 更新任务状态表和进度百分比
5. **报告生成**: 每天生成项目状态报告

#### 协调工具
- 共享状态文件中的"协调会议"章节
- 问题跟踪表格
- 依赖关系图

### 3. 作为开发Agent（database/backend/frontend/security）

#### 通用工作流程
1. **领取任务**: 查看`project_status.md`中的任务分配
2. **阅读指南**: 阅读对应的任务开发指南（`task_development_guides.md`）
3. **查看提示词**: 阅读专用的提示词工程文件（`prompt_*_agent.md`）
4. **开始开发**: 按照工作流程开始开发
5. **定期报告**: 每2-4小时在共享文件中报告进度
6. **提出问题**: 遇到问题立即在共享文件中提出
7. **完成任务**: 完成开发后更新状态，等待验收

#### 通信规范
每个Agent在共享文件中追加消息必须遵循格式：
```
### [时间] | [Agent名称]
**类型**: [进度报告|问题提出|接口发布|技术决策]
**任务**: [任务ID]
**进度**: X%
**内容**: [详细描述]
**问题**: [遇到的问题，如无则写"无"]
**下一步**: [下一步计划]
---
```

#### 特别注意事项
- **database Agent**: 必须先完成数据模型设计并发布，其他Agent才能开始相关工作
- **backend Agent**: 必须与frontend Agent确认每个接口的定义
- **frontend Agent**: 必须基于确认的接口定义开发，发现不一致立即提出
- **security Agent**: 必须等待backend基础框架完成后开始安全加固

### 4. 作为deployment_test Agent

#### 工作流程
1. **等待时机**: 等待主要开发任务基本完成后开始
2. **测试开发**: 开发测试套件，包括单元测试、集成测试、性能测试
3. **部署配置**: 配置Docker容器、Nginx、监控系统
4. **全面测试**: 在集成环境执行全面测试
5. **部署验证**: 验证部署配置，确保生产就绪

#### 测试报告
每次测试执行后必须在共享文件中报告：
- 测试通过率
- 测试覆盖率
- 发现的问题
- 性能测试结果

## 关键机制详解

### 1. 接口对齐机制

#### 流程
```
需求分析 → 接口设计 → 前后端确认 → 记录在案 → 并行开发 → 定期检查
```

#### 方法
1. **设计阶段**: backend Agent设计接口草案
2. **确认阶段**: frontend Agent确认接口草案，提出修改意见
3. **记录阶段**: 双方确认后，integration Agent记录到共享文件
4. **开发阶段**: 双方基于确认的接口开发，使用Mock数据
5. **检查阶段**: integration Agent定期检查实现与定义是否一致

#### 变更管理
任何接口变更必须：
1. 提出变更请求，说明原因和影响
2. 相关Agent评估影响
3. integration Agent协调决策
4. 记录变更，更新文档
5. 通知所有受影响方

### 2. 问题解决机制

#### 问题分类
- **P0紧急**: 系统崩溃、数据丢失、安全漏洞 - 立即处理
- **P1高**: 核心功能不可用、性能严重下降 - 4小时内处理
- **P2中**: 非核心功能问题、界面瑕疵 - 24小时内处理
- **P3低**: 优化建议、文档问题 - 下次迭代处理

#### 解决流程
```
发现问题 → 记录问题 → 分析原因 → 讨论方案 → 执行解决 → 验证关闭
    ↑          ↓          ↓          ↓          ↓          ↓
    └──────────┴──────────┴──────────┴──────────┴──────────┘
                 integration Agent协调跟踪
```

### 3. 进度监控机制

#### 监控维度
1. **时间进度**: 实际进度 vs 计划进度
2. **质量进度**: 测试通过率、代码覆盖率
3. **风险进度**: 风险识别和应对情况
4. **集成进度**: 接口一致性、模块集成情况

#### 监控工具
- 任务状态表（`project_status.md`中）
- 燃尽图（可由integration Agent维护）
- 里程碑跟踪
- 依赖关系图

#### 进度报告
- **每日**: 各Agent报告昨日完成、今日计划、问题阻塞
- **每周**: integration Agent生成周报，总结进展、问题、风险
- **里程碑**: 每个里程碑完成后全面评估

## 常见场景处理

### 场景1：Agent进度延迟
**症状**: 某个Agent进度明显落后于计划
**处理**:
1. integration Agent分析延迟原因
2. 如果是技术问题，组织相关Agent协助解决
3. 如果是资源问题，协调增加资源
4. 如果是需求问题，请求用户澄清
5. 调整依赖任务计划，尽量减少影响

### 场景2：接口不一致
**症状**: 前后端对接口理解不一致，集成失败
**处理**:
1. integration Agent组织前后端Agent会议
2. 对照最初接口定义，找出分歧点
3. 讨论解决方案，记录决策
4. 更新接口定义，相关Agent修改实现
5. 加强后续接口变更管理

### 场景3：技术方案争议
**症状**: 多个Agent对技术方案有不同意见
**处理**:
1. integration Agent组织技术讨论会
2. 各方案提出者陈述方案优劣
3. 基于数据（性能、成本、维护性）评估
4. 如无法达成共识，提交用户决策
5. 记录决策过程和最终方案

### 场景4：需求变更
**症状**: 用户提出新的需求或修改现有需求
**处理**:
1. 评估变更影响范围和工作量
2. 分析对当前进度的影响
3. 提出变更实施方案和时间调整
4. 用户批准后执行变更
5. 更新所有相关文档和计划
