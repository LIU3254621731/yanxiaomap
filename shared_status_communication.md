# 基于共享Markdown文件的Agent通信机制

## 概述
本方案采用一个共享的Markdown文件作为所有Agent之间的主要通信渠道。每个Agent定期在该文件中追加状态报告、提出问题、回答其他Agent的问题或向用户寻求澄清。这种方式直观、易读，适合小型到中型项目，便于人工监督和干预。

## Agent通信规则

### 1. 消息格式
每个Agent追加消息时必须遵循以下格式：

```
### [时间] | [Agent名称]
**类型**：[进度报告|问题提出|问题回答|接口定义|状态更新]
**相关任务**：[任务ID]
[内容详情]
---
```

### 2. 更新频率
- **进度报告**：每2小时至少一次，或任务状态有重大变化时
- **问题提出**：遇到阻塞时立即提出
- **问题回答**：看到相关问题时24小时内必须回答
- **接口定义**：接口有变更时立即更新

### 3. 问题解决流程
1. Agent遇到问题 → 在共享文件中提出问题
2. 其他Agent看到问题 → 在24小时内回答或转发
3. 如果问题涉及产品决策 → 标记为"需要用户输入"
4. 用户在"用户输入区"提供决策
5. 提问Agent根据决策继续工作

### 4. 冲突解决
1. 发现接口不一致 → 在共享文件中标记冲突
2. 相关Agent讨论解决方案
3. 如果无法达成一致 → 提交给integration Agent仲裁
4. integration Agent分析后提供建议方案
5. 相关Agent执行解决方案

### 5. 状态同步
1. 每个Agent完成任务里程碑时更新状态表
2. integration Agent定期检查状态一致性
3. 发现状态不一致时发起协调会议

## 文件维护

### 自动化工具
可以编写简单的脚本协助管理：

```bash
# 1. 添加新消息
./add_message.sh "frontend" "进度报告" "T001" "内容..."

# 2. 提取未解决问题
grep -n "问题：" project_status.md

# 3. 生成状态摘要
./generate_summary.sh project_status.md
```

### 版本控制
- 共享文件纳入Git版本控制
- 每次更新提交有意义的提交信息
- 定期备份历史版本

## 优势

1. **透明性**：所有通信记录对团队完全可见
2. **可追溯性**：完整的问题解决历史记录
3. **低技术门槛**：只需文本编辑器和Markdown知识
4. **灵活性**：支持结构化数据和自由文本混合
5. **人工友好**：便于项目负责人快速了解项目状态

## 扩展建议

### 小型项目（1-3个Agent）
- 直接使用单个Markdown文件
- 人工监控和协调

### 中型项目（3-7个Agent）  
- 使用主文件+按模块分页
- 添加自动化检查脚本
- 设置每日状态摘要邮件

### 大型项目（7个以上Agent）
- 考虑结合API通信自动同步到Markdown
- 使用GitHub Issues或Jira作为问题跟踪
- 保持Markdown作为人工查看的友好界面
