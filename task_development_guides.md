# 任务开发使用指南

## 概述
本文档为考研院校地图择校网站的各个开发任务提供详细的使用指南，包括任务目标、实施步骤、验收标准、通信要求和常见问题。每个任务对应一个Agent，请负责Agent严格按照指南执行。

## 通用开发规范

### 代码规范
1. **命名规范**: 使用有意义的英文命名，变量/函数使用camelCase，类名使用PascalCase，常量使用UPPER_SNAKE_CASE
2. **注释要求**: 所有公共函数、复杂逻辑必须添加中文注释，关键算法需要说明思路
3. **模块化**: 功能模块化，单一职责，避免过长函数和类
4. **错误处理**: 所有可能出错的地方必须有错误处理，避免程序崩溃
5. **日志记录**: 关键操作需要记录日志，便于调试和审计

### 版本控制
1. **分支策略**: 每个任务使用独立分支，命名格式: `feature/T001-database-design`
2. **提交信息**: 使用规范提交信息，格式: `类型(模块): 描述`，如 `feat(map): 实现地图点位渲染`
3. **代码审查**: 完成后提交Pull Request，由集成协调器或相关Agent审查

### 通信纪律
1. **进度报告**: 每2小时在`project_status.md`中更新进度
2. **问题提出**: 遇到阻塞立即在共享文件中提出问题，标注优先级
3. **接口变更**: 任何接口变更必须提前通知相关Agent，更新接口规范
4. **决策记录**: 所有技术决策记录在共享文件中，避免后续争议

## T001: 数据库设计与初始化

### 任务目标
根据需求文档中的9张表设计，创建完整的MySQL数据库结构，并导入初始数据。

### 实施步骤

#### 阶段1: 数据库设计审查（D1）
1. 仔细阅读需求文档中3.1-3.6节的表结构设计
2. 确认字段类型、长度、约束是否合理
3. 设计索引策略：主键索引、外键索引、查询频繁字段索引
4. 编写数据库设计文档，记录设计决策和理由

#### 阶段2: SQL脚本编写（D2）
1. 创建建表SQL脚本，包含所有表、字段、约束、索引
2. 创建外键关系脚本
3. 编写初始化数据SQL（院校、学科门类、一级学科、专业基础数据）
4. 创建数据库用户和权限脚本

#### 阶段3: 实体类生成（D2）
1. 根据表结构生成TypeORM实体类（如后续使用MyBatis-Plus，则生成实体类和Mapper）
2. 添加JPA注解或MyBatis-Plus注解
3. 创建基础Repository/DAO接口

#### 阶段4: 数据导入与验证（D3）
1. 准备初始数据Excel文件，转换为SQL插入语句
2. 执行完整SQL脚本，创建数据库
3. 验证数据完整性：表数量、记录数、外键关系
4. 进行基础查询性能测试

### 验收标准
1. ✅ 9张表完全按照需求文档设计创建
2. ✅ 所有字段类型、长度、约束正确
3. ✅ 索引创建合理，覆盖常用查询条件
4. ✅ 外键关系正确，级联操作配置合理
5. ✅ 初始数据导入完成，包含至少100所院校、20个学科门类、50个一级学科、200个专业
6. ✅ 实体类生成完整，与数据库表一一对应
7. ✅ 基础CRUD操作可通过实体类正常执行

### 通信要求
1. 数据库设计完成后，立即在共享文件中发布数据模型定义
2. 遇到字段歧义时，及时向integration Agent提问
3. 初始数据准备进度每4小时报告一次
4. 完成后通知backend和frontend Agent可以开始基于数据模型开发

### 交付物
1. `database/schema.sql` - 完整建表脚本
2. `database/init_data.sql` - 初始数据脚本
3. `database/entities/` - 实体类目录
4. `database/design_document.md` - 设计文档
5. `database/validation_report.md` - 验证报告

### 常见问题
1. **Q**: 字段长度不够怎么办？
   **A**: 参考需求文档中的示例数据，确保长度足够，如有疑问提出讨论

2. **Q**: 索引应该建哪些？
   **A**: 主键必建，外键必建，经常用于查询条件的字段（如院校名称、专业名称）建普通索引

3. **Q**: 初始数据从哪里来？
   **A**: 使用项目根目录下`mydate17_24/excel/`中的Excel文件，需要清洗转换

1. 配置SSL证书，实现HTTPS访问
2. 前端集成备案信息展示（ICP备案、公安网安备案）
3. 添加隐私政策页面
4. 配置数据脱敏（敏感数据不完整显示）

### 验收标准
1. ✅ 密码加密存储，不可逆
2. ✅ 图形验证码有效防止自动化攻击
3. ✅ 接口限流有效，防止恶意请求

## 紧急问题处理流程

### 问题分级
1. **P0紧急**: 系统崩溃、数据丢失、安全漏洞 - 立即处理
2. **P1高**: 核心功能不可用、性能严重下降 - 4小时内处理
3. **P2中**: 非核心功能问题、界面瑕疵 - 24小时内处理
4. **P3低**: 优化建议、文档问题 - 下次迭代处理

### 处理流程
1. **发现**: 在共享文件中报告问题，标注优先级
2. **分配**: integration Agent分配问题给负责Agent
3. **处理**: 负责Agent调查并解决问题
4. **验证**: test Agent验证问题是否解决
5. **关闭**: 问题解决后关闭，记录解决方案

3. **接口变更**: API接口参数或响应格式变化

### 变更流程
1. **提出**: 在共享文件中提出变更请求，说明原因和影响
2. **评估**: 相关Agent评估变更影响和工作量
3. **决策**: integration Agent协调决策，记录决策
4. **实施**: 各Agent按计划实施变更
5. **验证**: 验证变更效果，更新文档
