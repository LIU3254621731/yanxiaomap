## 严格限制

### 测试限制
1. **测试覆盖率**: 单元测试覆盖率必须>70%，核心业务代码覆盖率必须>80%
2. **测试通过率**: 所有测试必须100%通过，不得有关键测试跳过
3. **测试类型**: 必须包含单元测试、集成测试、API测试、性能测试
4. **测试环境**: 必须使用独立的测试环境，与开发、生产环境隔离
5. **测试数据**: 必须使用专用测试数据，不得污染生产数据

### 部署限制
1. **容器化**: 必须使用Docker容器化部署，不得直接部署到物理机
2. **编排工具**: 必须使用docker-compose，后期可扩展K8s
3. **环境分离**: 必须支持多环境（dev, test, prod）配置分离
4. **安全部署**: 必须使用非root用户运行容器，限制容器权限
5. **资源限制**: 必须设置容器资源限制（CPU、内存）
6. **健康检查**: 必须配置容器健康检查，支持自动恢复

### 监控限制
1. **监控覆盖**: 必须监控应用性能、服务器资源、数据库状态
2. **告警配置**: 必须配置关键指标告警（CPU>80%、内存>90%、服务宕机）
3. **日志收集**: 必须集中收集和分析日志
4. **备份监控**: 必须监控备份任务执行状态

### 性能限制
1. **响应时间**: 生产环境API响应时间必须<500ms（95分位）
2. **可用性**: 系统可用性必须>99.5%
3. **并发能力**: 必须支持至少100并发用户
4. **恢复时间**: 故障恢复时间必须<5分钟

### 通信纪律限制
1. **不确定必须与用户讨论**: 遇到任何不确定事项（测试策略、部署方案、监控配置等）必须立即在`project_status.md`中提出，等待用户明确答复后才能继续。

5. **测试协调**: 测试计划和结果必须通过`project_status.md`与其他Agent共享，确保问题及时修复。

## 问题处理

### 测试问题处理
1. **测试失败**: 分析失败原因，如果是bug则报告给相关Agent修复
2. **性能不达标**: 分析性能瓶颈，提出优化建议
3. **环境问题**: 解决测试环境配置问题

### 部署问题处理
1. **容器构建失败**: 分析Dockerfile问题，修复构建脚本
2. **服务启动失败**: 分析日志，解决配置或依赖问题
3. **网络问题**: 解决容器网络配置问题

### 监控问题处理
1. **数据采集失败**: 检查Prometheus配置和数据源
2. **告警不准确**: 调整告警阈值和规则
3. **仪表板问题**: 修复Grafana仪表板配置
