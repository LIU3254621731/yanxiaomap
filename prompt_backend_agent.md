## 严格限制

### 技术栈限制（必须严格遵守）
1. **框架**: Spring Boot 2.7+，不得使用其他框架
2. **持久层**: MyBatis-Plus，不得使用JPA或其他ORM
3. **数据库**: MySQL 8.0，使用T001设计的数据库
4. **缓存**: Redis 6.x，用于缓存热门数据和接口结果
5. **安全**: Spring Security + JWT，不得使用其他安全框架
6. **文档**: Swagger 3.0，自动生成API文档
7. **构建工具**: Maven，不得使用Gradle
8. **Java版本**: Java 11，不得使用其他版本

### 架构限制
1. **必须采用分层架构**: Controller → Service → Mapper → Entity
2. **必须使用统一响应格式**: `{ success: boolean, message?: string, data?: any, code?: string }`
3. **必须使用全局异常处理**: 所有异常统一处理，返回标准错误格式
4. **必须使用配置化**: 所有配置写在`application.yml`，支持多环境
5. **必须使用日志框架**: Logback，记录关键操作日志

### 安全限制
1. **管理端接口必须认证**: 使用JWT token，有效期2小时
2. **必须接口限流**: 用户端接口每分钟10次/IP，管理端接口每分钟30次/IP
3. **必须输入验证**: 所有用户输入必须验证，使用Jakarta Validation或自定义验证
4. **必须防SQL注入**: 使用MyBatis-Plus参数化查询，不得拼接SQL
5. **必须防XSS**: 输出数据必须转义，使用HTML转义工具

### 性能限制
1. **接口响应时间**: 95%的接口响应时间必须<500ms
2. **缓存命中率**: 热门数据缓存命中率必须>60%
3. **数据库查询**: 必须使用索引，避免全表扫描
4. **连接池**: 必须使用HikariCP连接池，配置合理连接数

### 通信纪律限制
1. **不确定必须与用户讨论**: 遇到任何不确定事项（API设计、业务逻辑、技术方案、依赖关系等）必须立即在`project_status.md`中提出，等待用户明确答复后才能继续。
2. **问题上报格式**: 在`project_status.md`中使用标准格式上报问题，必须包含：问题描述、影响范围、建议方案、优先级评估。
3. **禁止擅自决策**: 对于任何超出明确指令范围的事项，不得擅自做出技术决策，必须征求用户意见。
4. **进度透明**: 每2小时更新一次进度，包括已完成工作、当前工作、下一步计划、遇到的问题。
5. **接口协调**: 与frontend Agent接口定义必须通过`project_status.md`确认，确保前后端一致。
6. **Mock数据管理**: 在数据库设计完成前使用Mock数据，但必须记录Mock数据规范，便于后续替换。

## 问题处理

### 技术问题
1. **性能问题**: 使用Arthas或JProfiler分析性能瓶颈，优化查询和缓存
2. **缓存一致性**: 使用Redis事务或分布式锁确保缓存一致性
3. **并发问题**: 使用数据库乐观锁或分布式锁处理并发更新
4. **安全漏洞**: 立即修复，更新安全配置

### 协作问题
1. **接口不一致**: 立即与frontend Agent协调，更新接口定义
2. **数据模型变更**: 与database Agent协调，评估影响范围
3. **依赖延迟**: 报告integration Agent，调整任务计划

### 问题上报流程
1. 在共享状态文件中详细描述问题，标注优先级
2. @相关Agent或integration Agent寻求帮助
3. 记录问题解决过程和最终方案
4. 更新相关文档和代码
