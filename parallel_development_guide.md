# 多Agent并行开发指南

## 概述

本文档详细说明如何在考研院校地图择校网站项目中实现多个Agent同时开发，确保并行开发效率最大化，同时保持系统一致性和质量。

## 并行开发策略

### 核心原则

1. **接口先行**：基于`project_status.md`中的接口规范开始开发，不等待具体实现
2. **Mock数据驱动**：在依赖组件完成前使用Mock数据进行开发
3. **持续协调**：通过`project_status.md`持续协调接口和数据模型
4. **用户决策中心**：所有不确定事项必须由用户决策

### 各Agent并行工作安排

| Agent | 任务 | 并行开始时间 | 依赖处理 | 交付物 |
|-------|------|-------------|----------|--------|
| **database** | T001 数据库设计与初始化 | Day 1 | 无 | 数据库schema、实体类、初始数据 |
| **backend** | T002 后端核心API开发 | Day 1 (并行) | 基于接口规范，数据库操作层暂用Mock | Controller、Service层、Mock数据层 |
| **frontend** | T003 前端用户端开发 | Day 1 (并行) | 基于接口规范，API调用暂用Mock数据 | UI组件、地图集成、Mock数据适配器 |
| **backend** | T004 后端管理端开发 | 等待T002基础框架 | 依赖T002 | 管理端CRUD接口、权限控制 |
| **security** | T005 安全合规实现 | 等待T002基础框架 | 依赖T002 | 安全加固、合规集成 |
| **deployment+test** | T006 集成测试与部署 | 等待T001-T005基本完成 | 依赖所有任务 | 测试套件、Docker部署、监控 |
| **integration** | T007 项目集成与协调 | 全程 | 监控所有任务 | 协调报告、集成计划 |

## Mock数据管理规范

### Mock数据设计原则

1. **真实性**：Mock数据应尽可能接近真实数据结构
2. **一致性**：前后端Mock数据必须保持一致
3. **可替换性**：Mock数据层应易于替换为真实实现
4. **版本控制**：Mock数据应与接口规范版本保持一致

### 后端Mock数据实现

```java
// 示例：在数据库层完成前使用Mock
@Service
public class SchoolServiceMock implements SchoolService {
    @Override
    public List<SchoolDTO> getSchools(MapFilter filter) {
        // 返回符合接口规范的Mock数据
        return List.of(
            new SchoolDTO(1, "清华大学", 116.327, 39.999, "985", "北京", "北京",
                List.of(new MajorDTO("计算机科学与技术", 2024, 385))),
            new SchoolDTO(2, "北京大学", 116.316, 39.999, "985", "北京", "北京",
                List.of(new MajorDTO("软件工程", 2024, 380)))
        );
    }
}
```

## 接口协调机制

### 接口定义流程

1. **初步定义**：在`project_status.md`的"接口规范"章节定义初始接口
2. **前端确认**：frontend Agent确认接口满足前端需求
3. **后端确认**：backend Agent确认接口技术可行性
4. **用户批准**：用户最终批准接口定义
5. **版本管理**：接口变更必须记录版本和变更原因

### 接口一致性检查表

每次前后端Agent更新代码时，必须检查：
- [ ] 请求路径和方法是否一致
- [ ] 请求参数名称和类型是否一致  
- [ ] 响应数据结构是否一致
- [ ] 错误处理方式是否一致
- [ ] 分页参数约定是否一致

## 通信纪律执行细则

### 问题上报标准格式

```
### [时间] | [Agent名称]
**类型**: [问题/决策需求/进度报告]
**任务**: [任务ID]
**优先级**: [高/中/低]
**问题描述**: [清晰描述问题]
**影响范围**: [影响哪些任务或功能]
**建议方案**: [提出1-3个解决方案]
**用户决策需求**: [需要用户决策的具体事项]
```

### 进度报告标准格式

```
### [时间] | [Agent名称]
**类型**: 进度报告
**任务**: [任务ID]
**进度**: [百分比]
**已完成**: [具体完成的工作]
**当前工作**: [正在进行的工怍]
**下一步计划**: [接下来要做的工怍]
**问题/阻塞**: [遇到的问题或阻塞]
**预计完成**: [预计完成时间]
```

### 违反通信纪律的处理

1. **首次违反**：integration Agent发出警告
2. **再次违反**：在`project_status.md`中公示违规记录
3. **严重违规**：暂停任务执行，等待用户处理

## 依赖关系管理

### 关键依赖路径

```
T001(数据库) → T002(后端API) → T004(管理端)
                    ↓
              T003(前端) → T006(集成测试)
                    ↓
              T005(安全) → T006(集成测试)
```

### 并行开发依赖解决方案

| 依赖 | 解决方案 | 责任人 |
|------|----------|--------|
| T002依赖T001 | 使用Mock数据层，数据库设计完成后替换 | backend Agent |
| T003依赖T001 | 使用Mock数据结构，数据库设计完成后更新 | frontend Agent |
| T003依赖T002 | 使用Mock API调用，API实现完成后替换 | frontend Agent |
| T004依赖T002 | 等待T002基础框架完成 | backend Agent |
| T005依赖T002 | 等待T002基础框架完成 | security Agent |

## 集成时间表

### 第一阶段：并行开发期（第1-3天）
- database Agent：完成数据库设计
- backend Agent：完成Controller和Service层框架
- frontend Agent：完成UI组件和地图集成框架
- integration Agent：监控接口一致性和Mock数据协调

### 第二阶段：集成准备期（第4-5天）
- database Agent：提供最终数据库脚本和实体类
- backend Agent：替换Mock数据层为真实数据库访问
- frontend Agent：替换Mock API调用为真实API调用
- integration Agent：组织集成测试准备

### 第三阶段：完整集成期（第6-7天）
- backend Agent：完成T004管理端开发
- security Agent：完成安全加固
- deployment+test Agent：执行集成测试和部署
- integration Agent：协调最终集成和用户验收

## 风险管理

### 并行开发风险及应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 接口不一致 | 中 | 高 | 每日接口一致性检查，integration Agent协调 |
| Mock数据与真实数据差异 | 中 | 中 | 建立Mock数据验证机制，定期与database Agent对齐 |
| 进度不同步 | 低 | 高 | 每小时进度监控，及时调整任务分配 |
| 技术决策冲突 | 低 | 中 | 用户最终决策机制，记录决策过程 |
| 沟通延迟 | 低 | 低 | 强制2小时进度更新，问题4小时内响应 |

### 风险升级机制

1. **Level 1**：Agent内部解决，记录在`project_status.md`
2. **Level 2**：integration Agent协调解决
3. **Level 3**：用户决策解决
4. **紧急风险**：立即通知用户，暂停相关任务

## 质量保证

### 并行开发质量标准

1. **代码质量**：即使使用Mock数据，代码也必须符合编码规范
2. **测试覆盖**：Mock实现也应有单元测试
3. **文档完整**：所有Mock数据和接口必须有文档说明
4. **可维护性**：Mock层应易于移除和替换

### 验收检查点

每个阶段结束时必须通过以下检查：
- [ ] 所有接口定义在`project_status.md`中确认
- [ ] Mock数据与接口规范一致
- [ ] 代码通过基础语法检查
- [ ] 进度报告按时提交
- [ ] 无未解决的高优先级问题

## 用户参与指南

### 用户需要参与的决策点

1. **接口最终确认**：前后端Agent协商后的接口定义
2. **技术方案选择**：当有多个可行技术方案时
3. **优先级调整**：当任务进度出现偏差时
4. **风险评估**：对识别出的重大风险
5. **验收标准**：每个阶段的验收标准

### 用户决策响应时间承诺

- **紧急决策**：2小时内响应
- **高优先级决策**：4小时内响应  
- **中优先级决策**：8小时内响应
- **低优先级决策**：24小时内响应

## 总结

通过本文档定义的并行开发机制，多个Agent可以同时高效工作，同时确保最终系统的一致性和质量。关键成功因素包括：

1. **严格的通信纪律**：不确定必须与用户讨论
2. **完善的Mock数据管理**：确保并行开发可行性
3. **持续的接口协调**：避免接口不一致
4. **主动的风险管理**：及时发现和解决问题
5. **明确的用户参与**：确保项目方向正确

所有Agent必须严格遵守本指南，integration Agent负责监督执行。
