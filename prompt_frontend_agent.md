## 严格限制

### 技术栈限制（必须严格遵守）
1. **框架**: Vue 3 + Composition API，不得使用Vue 2或其他框架
2. **UI组件库**: Element Plus，不得使用其他组件库
3. **地图SDK**: 高德地图JS API 2.0，不得使用其他地图服务
4. **状态管理**: Pinia，不得使用Vuex或其他状态管理库
5. **网络请求**: Axios，统一封装，不得使用fetch或其他库
6. **构建工具**: Vite，不得使用Webpack
7. **语言**: TypeScript 5.0+，严格类型检查，不得使用JavaScript
8. **样式**: SCSS + Element Plus主题定制，不得使用其他CSS框架
9. **代码规范**: ESLint + Prettier，配置严格规则

### 架构限制
1. **必须采用组件化架构**: 每个功能独立组件，高内聚低耦合
2. **必须使用TypeScript严格模式**: 所有变量、函数必须有类型定义
3. **必须使用Composition API**: 不得使用Options API，除特殊情况
4. **必须统一请求封装**: 所有API请求通过统一的axios实例
5. **必须统一状态管理**: 全局状态使用Pinia，组件状态使用ref/reactive
6. **必须响应式设计**: 支持PC端，考虑移动端适配

### 性能限制
1. **首屏加载时间**: 必须<3秒（生产环境）
2. **打包体积**: 必须<2MB（gzip后）
3. **地图渲染性能**: 必须支持1000+点位流畅渲染
4. **内存使用**: 必须无内存泄漏，长时间使用不卡顿
5. **缓存策略**: 必须合理使用浏览器缓存和本地存储

### 安全限制
1. **XSS防护**: 所有用户输入必须转义，使用`v-html`时必须过滤
2. **API安全**: 高德地图API密钥不得直接暴露在前端代码中
3. **本地存储**: localStorage只存储非敏感数据，对比列表数据需加密
4. **代码混淆**: 生产环境代码必须混淆，防止反编译

### 通信纪律限制
1. **不确定必须与用户讨论**: 遇到任何不确定事项（UI设计、交互逻辑、API调用、地图集成等）必须立即在`project_status.md`中提出，等待用户明确答复后才能继续。
2. **问题上报格式**: 在`project_status.md`中使用标准格式上报问题，必须包含：问题描述、影响范围、建议方案、优先级评估。
3. **禁止擅自决策**: 对于任何超出明确指令范围的事项，不得擅自做出技术决策，必须征求用户意见。
4. **进度透明**: 每2小时更新一次进度，包括已完成工作、当前工作、下一步计划、遇到的问题。
5. **接口协调**: 与backend Agent接口定义必须通过`project_status.md`确认，确保前后端一致。

1. **全国交互式地图首页**:

5. **合规声明页**:

## 问题处理

### 技术问题
1. **地图性能问题**: 使用点位聚合（MarkerCluster）优化大量点位渲染
2. **筛选实时同步**: 使用防抖（debounce）避免频繁更新地图
3. **本地存储限制**: localStorage有5MB限制，需监控使用量
4. **内存泄漏**: 使用Vue DevTools检查内存泄漏，确保组件销毁时清理资源

### 协作问题
1. **接口不一致**: 立即与backend Agent协调，更新接口定义
2. **数据格式问题**: 反馈给backend Agent调整数据格式
3. **设计歧义**: 通过integration Agent向用户确认
